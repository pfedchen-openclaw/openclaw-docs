### P6-23 — Bind source is captured at container creation; fix-after-create needs recreate
**Type:** finding / lesson · **Opened+closed:** 6c.1
Two compounding traps cost a detour. (1) `gmail_helper.py` had been deployed to `agent-creds/` as an **empty directory**, not the script — a stray `docker run -v` against a not-yet-present source makes Docker auto-create an empty dir **at the host source**, and the later copy didn't overwrite it. **Lesson:** a file destined for a bind mount must be verified as a **file** (`[ -f ]` + sha + byte length), not merely path-exists, before binding. (2) A bind mount captures the host path's **type at container-creation time**; replacing the host dir with a file afterward is invisible to an already-running container. Marie's `pa` container was created during the empty-dir window (`recreate --all` ran *before* the host fix), so it carried a stale empty-dir mount and reported `/opt/gmail_helper.py` as a directory — correctly. **Fix:** `openclaw sandbox recreate --agent <id> --force` **after** the host source is correct. **Ordering rule: recreate comes after the host fix, never before.** virtiofs single-file binds work fine once the source is a real file (the "empty dir" was never a virtiofs limitation). `recreate` is lazy (removes; respawns on next agent turn) — confirm via a real turn, not just `inspect`. (Also: a token-endpoint `GET /` returning HTTP 404 is **egress success**, not a block — a blocked container throws a DNS/connection/timeout error, never a clean 404.)

### P6-35 — SECURITY: sandbox egress reaches the host gateway control-plane + SSH via host.lima.internal (decision surfaced)
**Type:** security finding + decision · **Opened:** 6c.4 (2026-06-12) · **Status:** OPEN — boundary holds (token-gated); browser-level guard shipped; host-level hardening surfaced as a decision.
- **Finding.** From inside the `mode:all` sandbox (`network: bridge`) the container reaches **host loopback** via lima's `host.lima.internal` = **192.168.5.2** (gvproxy forwards guest→host-loopback). Reachable: **`192.168.5.2:18789`** → the OpenClaw **gateway control plane** (healthz 200 + "OpenClaw Control" UI HTML) and **`192.168.5.2:22`** → host SSH. Blocked (good): cloud metadata `169.254.169.254`, container-loopback `127.0.0.1:18789`, sample private LAN. **Never characterised** in 6c.1 — the P6-21 decision opened `bridge` for outbound internet but did not consider host-loopback reachability. Predates 6c.4, but the **browser amplifies it** (web = highest injection surface; a malicious page could attempt SSRF to the gateway).
- **Why the boundary still holds.** `gateway.auth.mode = token`, `gateway.auth.token` set (21B), `controlUi.allowInsecureAuth = false`. The control API is **token-gated**; the token lives in `openclaw.json` — **outside the container** (binds are only gmail_helper/google.json/google_helper/workspace). The container can *reach* the gateway but **cannot drive it** without the token it does not hold. The Strategy-2 dead-end (secrets outside the container) is intact. *Caveat:* a full unauthenticated-endpoint audit was NOT done — only `/healthz` + `/` probed; mutating endpoints deliberately not poked.
- **Mitigation shipped (6c.4).** `browser_helper.py`'s **egress guard** refuses navigation + every sub-request to loopback/private/link-local/metadata/host targets via a Playwright route handler on the same proven `host_blocked` predicate. **Proven both ways:** top-level navigation refused (throwaway + both live containers + a real Charlotte turn), AND **programmatic SSRF blocked** — a loaded public page's `fetch('http://192.168.5.2:18789/healthz')` and `fetch('http://169.254.169.254/')` both return "Failed to fetch" (route-aborted). Closes the browser-as-SSRF vector; does **not** close raw socket access from arbitrary in-container code (that is P6-35's host-level decision).
- **DECISION SURFACED (Peter's call):** **(A) Accept** — token-gating + browser guard are the effective boundary; record + revisit. **(B) Harden the VM egress** — firewall rule in the Colima/Lima guest to DROP container→192.168.5.2 while preserving DNS (192.168.5.1) + public internet (defence-in-depth; mind persistence across colima restarts + not breaking egress). **Recommendation: (A) now + (B) as a scoped follow-up** — no live escape exists; (B) is a deliberate network change, not an SSH-session afterthought.
- **DECISION TAKEN (Peter, 6c.4 2026-06-12): defer to 6c.7.** Accept token-gating + the proven browser egress-guard as the boundary for now; the host-level VM egress hardening (option B — DROP container→192.168.5.2, preserve DNS + public internet, verify persistence across colima restarts) is **bundled into the 6c.7 ops & reliability/security pass**.
