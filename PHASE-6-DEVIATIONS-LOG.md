### P6-23 — Bind source is captured at container creation; fix-after-create needs recreate
**Type:** finding / lesson · **Opened+closed:** 6c.1
Two compounding traps cost a detour. (1) `gmail_helper.py` had been deployed to `agent-creds/` as an **empty directory**, not the script — a stray `docker run -v` against a not-yet-present source makes Docker auto-create an empty dir **at the host source**, and the later copy didn't overwrite it. **Lesson:** a file destined for a bind mount must be verified as a **file** (`[ -f ]` + sha + byte length), not merely path-exists, before binding. (2) A bind mount captures the host path's **type at container-creation time**; replacing the host dir with a file afterward is invisible to an already-running container. Marie's `pa` container was created during the empty-dir window (`recreate --all` ran *before* the host fix), so it carried a stale empty-dir mount and reported `/opt/gmail_helper.py` as a directory — correctly. **Fix:** `openclaw sandbox recreate --agent <id> --force` **after** the host source is correct. **Ordering rule: recreate comes after the host fix, never before.** virtiofs single-file binds work fine once the source is a real file (the "empty dir" was never a virtiofs limitation). `recreate` is lazy (removes; respawns on next agent turn) — confirm via a real turn, not just `inspect`. (Also: a token-endpoint `GET /` returning HTTP 404 is **egress success**, not a block — a blocked container throws a DNS/connection/timeout error, never a clean 404.)

### P6-42 — Renounce the host-browser domain allowlist → IP-class SSRF filtering (6c.7)
**Type:** decision + implementation (security posture) · **Opened+Resolved:** 6c.7 (2026-06-13) · **Decided by Peter.** Supersedes the allowlist-as-trust-routing framing of [P6-39]; dissolves [P6-41].

**Empirical guard analysis (A-12, read the 2026.4.22 build).** The native browser guard (`dist/cdp.helpers-*.js` → `ssrf-*.js`) has **two layers**: **Gate 1** (`isExplicitlyAllowedBrowserHostname`) is a DNS-rebinding pre-filter — for the host browser it treats an **empty** `hostnameAllowlist` as **deny-all** (require IP-literal), which is what forces enumeration. **Gate 2** (`resolvePinnedHostnameWithPolicy`) resolves the host and **blocks any private/internal/loopback/link-local/reserved IP independently of the allowlist** — this is the layer that actually protects the machine/LAN. **Change applied.** (1) Config `browser.ssrfPolicy.hostnameAllowlist = []` via `openclaw config set --strict-json` (gate-clear). (2) **One-token guard patch**: in `isExplicitlyAllowedBrowserHostname`, empty-list branch `… : false` → `… : true`, so an empty list **defers to Gate 2** instead of denying. `dangerouslyAllowPrivateNetwork` stays `false`. Net posture = **all PUBLIC hostnames allowed, all private/internal IPs blocked** — IP-class egress filtering, *strictly more secure* than the `dangerouslyAllowPrivateNetwork` escape hatch. Original guard backed up to `cdp.helpers-BWvLGafd.js.orig-6c7`; live guard sha `63e4284773ad`/22170B.
